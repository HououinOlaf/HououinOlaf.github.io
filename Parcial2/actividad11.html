# 01_EXECUTIVE_SUMMARY

> Evaluación: La máquina Napping 101 fue comprometida mediante una cadena de dos fallas conectadas. Primero, un flujo web roto donde contenido manipulado es procesado por un administrador; luego, una automatización local insegura que habilitó la escalada a root.

> Riesgo Ejecutivo (Crítico): Esta brecha permite acceso operativo pleno, pérdida de control del sistema y posible alteración o robo de archivos, comprometiendo al 100% la tríada CIA.

> Metodología: Ejecución estructurada aplicando PTES (Penetration Testing Execution Standard) apoyada con OWASP para la inspección de la capa de aplicación web.

# 02_BITÁCORA_DE_EXPLOTACIÓN_(KILL_CHAIN)

> Análisis detallado de la trazabilidad metodológica del ataque.

VECTORES ACTIVOSHTTP (80) + SSH (22)
FALLA INICIALSession Cookies sin HttpOnly

> Mapeo Inicial: Utilizando netdiscover y Nmap, se aisló el servidor (IP: 192.168.56.101). La presencia de SSH y HTTP indicó que cualquier debilidad web podría escalar a un acceso persistente a nivel de sistema operativo.

> Lógica de Negocio Quebrantada: La inspección del portal no se limitó a pruebas ciegas. Se analizó el flujo funcional: Un usuario se registra, inicia sesión y accede a un panel donde envía enlaces para que sean revisados por un administrador.

> Hipótesis: Este diseño convierte la revisión administrativa en una superficie de ataque. Un usuario sin privilegios puede introducir contenido hostil que terminará siendo procesado con alta confianza.

Recon Evidence
# Nmap reportó fallas de hardening en sesión PHP
PORT > 80/tcp open http (Apache 2.4.41)
FLAGS > PHPSESSID: httponly flag not set
# Flujo detectado en portal HTML
Web > "All links will be reviewed by our admin"
OBJETIVORobo de Sesión / Pivote
HALLAZGOAcceso útil concedido

> Procedimiento: Se suministró contenido controlado dentro del flujo de revisión administrativa. Dado que el portal confía ciegamente en el enlace proporcionado, el administrador procesa el input del atacante.

> Resultado: La interacción forzó la exposición de credenciales/sesiones válidas, otorgando al equipo de pentesting un acceso operativo legítimo al entorno web y, mediante reutilización, a la interfaz remota (SSH).

OBJETIVOControl total del host
VULNERABILIDADAbuso de Cronjob / Tareas

> Identificación del Riesgo: Una vez dentro del sistema, se identificó un problema grave de permisos. Una automatización o script crítico programado (Cronjob) era ejecutado constantemente por el usuario root, pero el archivo base era modificable por usuarios sin privilegios.

> Explotación Local: Se sobrescribió la ruta de ejecución de este archivo para inyectar una shell inversa. Cuando el servidor procesó su tarea automatizada, ejecutó la instrucción como superusuario.

> Impacto Final: Se obtuvo acceso interactivo de Root, validando el compromiso absoluto de la máquina mediante la captura de la flag: Admins just can't stay awake tsk tsk tsk.

# 03_ROADMAP_DE_REMEDIACIÓN

// CONTENCIÓN INMEDIATA (0-7 DÍAS)

> Hotfix de Escalada: Corregir permisos en archivos leídos por Cron (chmod 755 y chown root).

> Aislar revisión administrativa (usar Sandbox o deshabilitar temporalmente).

> Rotar todas las contraseñas web y llaves SSH comprometidas.

// HARDENING (8-30 DÍAS)

> Desinfectar la aplicación web garantizando validación de entradas estrictas (OWASP ASVS).

> Actualizar Apache y parametrizar cabeceras HTTP (HttpOnly/Secure).

> Endurecer el acceso SSH limitando IP y bloqueando contraseñas por defecto.

// MADUREZ Y GOBIERNO (30-90 DÍAS)

Se debe ejecutar un "Threat Modeling" de la lógica de negocio para futuros despliegues, estableciendo monitoreo continuo (SIEM) de logs críticos. La seguridad no se vulneró con un "golpe maestro", sino con el eslabonamiento de debilidades de la capa de aplicación hacia el sistema. La auditoría recurrente del Principio de Menor Privilegio (POLP) es obligatoria para garantizar la salud a largo plazo del activo.
