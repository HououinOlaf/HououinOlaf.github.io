# 01_MARCO_TEÓRICO:_SQL_INJECTION

> Definición del Riesgo: Las aplicaciones web modernas dependen ampliamente de bases de datos para almacenar información crítica (credenciales, datos financieros, registros). Esta interacción constante introduce vulnerabilidades graves cuando las entradas del usuario no son validadas adecuadamente.

> ¿Qué es SQL Injection (SQLi)? Es una vulnerabilidad que ocurre cuando una aplicación permite que datos maliciosos introducidos por un atacante se incorporen directamente dentro de consultas SQL ejecutadas por el backend. Esto altera la lógica original de la consulta.

> Impacto (CIA): El atacante puede evadir la autenticación, acceder a información sensible (Confidencialidad), modificar o borrar registros (Integridad) o incluso provocar la caída de la base de datos (Disponibilidad).

// CLASIFICACIÓN_DE_ATAQUES_SQLi

  • > In-band SQLi (Classic/Union): El atacante utiliza el mismo canal de comunicación para lanzar el ataque y extraer los resultados directamente en la respuesta HTTP.
  • > Inferential SQLi (Blind): No hay transferencia directa de datos. El atacante infiere la estructura y los datos observando el comportamiento del servidor (provocando errores forzados o retrasos de tiempo).
  • > Out-of-band SQLi (OAST): Los datos se exfiltran utilizando un canal alternativo, forzando al servidor de base de datos a emitir peticiones DNS o HTTP hacia un dominio controlado por el atacante.

TIPO DE SQLiSQLi clásica (Cláusula WHERE)

> Objetivos: Modificar la consulta SQL para que la aplicación muestre todos los productos de una categoría específica, incluyendo aquellos que aún no han sido lanzados (released = 0).

> Vulnerabilidad: La vulnerabilidad ocurre porque la aplicación concatena directamente la entrada del usuario en una consulta SQL sin sanitizarla. La consulta original es: SELECT * FROM products WHERE category = '[CATEGORÍA]' AND released = 1. Al no haber validación, un atacante puede cerrar la comilla simple y añadir operadores lógicos (OR) alterando la condición.

> Procedimiento:
1. Acceder al laboratorio e interactuar con el filtro seleccionando cualquier categoría (ej. "Gifts").
2. Abrir Burp Suite, interceptar la petición que se ve así: GET /filter?category=Gifts.
3. Enviar la petición al Repeater, cambiando el valor por el payload: ' OR 1=1--
4. Enviar y observar que la respuesta HTTP 200 contiene productos ocultos.

> Explicación del payload:
- ' (Comilla simple): Rompe la cadena de texto original en la consulta SQL.
- OR 1=1: Es una condición siempre verdadera. Basta con que sea cierto para devolver todas las filas.
- -- (Doble guion): Carácter de comentario en SQL que hace que el resto de la consulta original sea ignorado.

> Mitigación recomendada: La forma más efectiva de prevenir este ataque es dejar de concatenar strings y utilizar Consultas Preparadas (Parameterized Queries), pasando el valor del usuario como un parámetro aislado.

> Vulnerabilidad: La base de datos es susceptible a manipulación. Si se inyecta un UNION con distinto número de columnas, arroja un error visible como 'column count mismatch'.

TIPO DE SQLiBlind SQLi (Conditional Errors)

> Objetivos: Inferir la contraseña forzando errores intencionales en la BD mediante cálculos matemáticos imposibles (división por cero) condicionados a valores lógicos.

> Vulnerabilidad: A diferencia del anterior, aquí no hay "Welcome back". La página siempre es idéntica. Pero si la consulta causa un fallo interno (Error 500), se puede usar ese error como indicador booleano.

> Procedimiento:
1. Modificar la Cookie TrackingId insertando una sentencia `CASE WHEN`.

TIPO DE SQLiUNION-based (WAF Bypass)

> Objetivos: Evadir el Firewall de Aplicaciones Web (WAF) que bloquea firmas comunes de inyección como "UNION SELECT", ofuscando la carga útil.

> Vulnerabilidad: La característica 'stock check' transmite datos en XML. El analizador XML del backend decodifica entidades Hexadecimales ANTES de enviarlas al motor SQL, pero el WAF las evalúa tal como llegan.

> Explicación del payload:
- El tag engaña al WAF con ofuscación en Hex. El motor ensambla el UNION extrayendo el username y password por la tubería permitida.

> Mitigación recomendada: Reforzar el WAF para normalizar y decodificar XML antes del análisis de firmas (Deep packet inspection).
